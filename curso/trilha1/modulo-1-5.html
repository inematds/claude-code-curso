1

🧠 Como a memória funciona

🧠 Imagine assim: o Claude Code nao tem memória magica. Tudo que ele "lembra" vive no context window da sessão atual. E como uma mesa de trabalho: o que esta na mesa, ele ve. O que saiu, esqueceu.

A memória do Claude Code e fundamentalmente diferente da memória humana. Ela e efemera por sessão: tudo que voce disse, tudo que o agente leu, todas as respostas ficam no context window daquela sessão. Quando a sessão acaba, a "memoria" tambem acaba.

O que persiste entre sessões sao: o CLAUDE.md (lido automaticamente), notas salvas com /memory, e o proprio codebase (que o agente pode re-ler). Nenhum desses e memória do modelo. Sao arquivos que o modelo le de novo em cada sessão.

Em 1 frase: context window = mesa de trabalho temporaria; CLAUDE.md e /memory = notas persistentes.

2

💡 O comando /memory

🧠 Imagine assim: o /memory e um atalho para salvar preferências rapidas sem editar CLAUDE.md. E como colar um post-it na tela do computador que persiste entre sessões.

Dentro de uma sessão, digitar /memory abre um editor para salvar notas que o Claude Code vai ler automaticamente em todas as sessões futuras daquele projeto. Internamente, essas notas sao armazenadas no CLAUDE.md gerado.

Use /memory para preferências rapidas: "prefiro respostas em português", "sempre rode testes antes de commitar", "o banco de dados de dev esta em localhost:5432". Para instruções mais complexas e estruturadas, edite o CLAUDE.md diretamente.

Em 1 frase: /memory salva notas rapidas que persistem entre sessões sem editar CLAUDE.md manualmente.

3

📐 Context window — o recurso escasso

🧠 Imagine assim: pense no context window como a RAM do Claude. Tem um limite fixo. Quando voce enche com arquivos e conversa, o desempenho cai. Gerenciar contexto e uma habilidade critica.

O context window tem um limite fixo de aproximadamente 200k tokens. Tudo compete por esse espaço: o system prompt (CLAUDE.md), suas mensagens, os arquivos que o agente le, as respostas geradas, e o historico da conversa. Conforme a sessão cresce, o espaço disponivel diminui.

Na pratica, uma sessão produtiva dura entre 30 minutos e 2 horas antes de o contexto começar a encher. Sessões mais longas sofrem de degradação gradual: o modelo começa a esquecer detalhes do inicio da conversa, repetir coisas, e perder o fio da meada.

⚠️ Sinais de contexto cheio

O Claude começa a esquecer instruções do inicio, repete perguntas ja respondidas, ignora regras do CLAUDE.md, ou simplesmente para de funcionar bem. Quando perceber isso, e hora de compactar ou iniciar sessão nova.

Em 1 frase: 200k tokens = limite fixo; sessões longas degradam. Gerencie o contexto ativamente.

4

🗜️ Compaction — quando e como

🧠 Imagine assim: compaction e o "resumão" automatico. Quando o contexto enche, o Claude Code comprime a conversa num resumo menor. Ganha espaço, mas perde detalhes.

A compaction acontece de duas formas: automatica (quando o contexto atinge ~95% do limite) ou manual (via o comando /compact). O processo resume toda a conversa ate aquele ponto num texto menor, liberando espaço.

O trade-off e claro: voce ganha espaço mas perde nuances. Detalhes especificos, decisões de codigo, explicações longas podem ser resumidas de forma imprecisa. Por isso e importante fazer checkpoints (commitar, salvar notas) antes que a compaction aconteça.

compactação manual
# compactar manualmente (dentro da sessão)
/compact

# o Claude resume a conversa e libera espaço
# use ANTES de o contexto encher pra ter controle

Em 1 frase: /compact resume a conversa para liberar espaço; perde detalhes, entao faça checkpoints antes.

5

🔗 Session handoff entre conversas

🧠 Imagine assim: quando uma sessão fica longa demais, e melhor iniciar uma nova e "passar o bastao". A arte esta em transferir o contexto essencial sem arrastar lixo.

Existem tres estratégias para handoff entre sessões: (1) /resume retoma a sessão anterior como estava. (2) Iniciar sessão nova com um resumo manual: peça ao Claude "resuma o que fizemos ate agora em 10 bullets" e cole isso na sessão nova. (3) Salvar o estado no CLAUDE.md ou num arquivo de notas.

A melhor pratica e combinar: commite o codigo (checkpoint no git), atualize o CLAUDE.md se algo importante mudou, e inicie sessão nova. Assim o agente começa fresco mas com todo o contexto necessario nos arquivos.

Em 1 frase: commite o codigo + atualize o CLAUDE.md + inicie sessão nova = handoff limpo.

6

📍 Estratégias de checkpoint

🧠 Imagine assim: checkpoints previnem perda de progresso. Git commit e o checkpoint natural. Combine com notas em CLAUDE.md e TODO files para tarefas longas.

O checkpoint mais natural no desenvolvimento e o git commit. A cada bloco significativo de trabalho, commite. Se o contexto encher ou a sessão degradar, voce sempre pode iniciar uma sessão nova no commit mais recente sem perder nada.

Para tarefas muito longas (horas de trabalho), adicione checkpoints intermediarios: um arquivo TODO.md com o estado atual, notas no CLAUDE.md sobre decisões tomadas, e commits parciais com mensagens descritivas. Quando a nova sessão começar, o agente le esses arquivos e retoma o trabalho.

estrategia-checkpoint.txt
Tarefa longa? Faça checkpoints:
1. git commit a cada bloco de trabalho
2. Atualize CLAUDE.md se decisões mudaram
3. Use /memory pra notas rápidas
4. Peça ao Claude "resuma o estado atual"
5. Ao iniciar sessão nova, o contexto está nos arquivos

Em 1 frase: git commit = checkpoint natural; combine com CLAUDE.md e TODO.md para tarefas longas.

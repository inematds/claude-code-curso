1

Commits semanticos com Claude

Quando voce pede ao Claude para commitar, ele analisa o diff, identifica o tipo de mudanca (feat, fix, refactor, docs, etc.) e gera uma mensagem no padrao conventional commits. Isso mantem o historico legivel e facilita changelogs automaticos.

commit-semantico.sh
# Peca ao Claude para commitar as mudancas atuais
# Ele vai:
# 1. Rodar git status e git diff
# 2. Analisar o que mudou
# 3. Gerar mensagem semantica
# 4. Commitar com a mensagem

# Prompt: "commita essas mudancas com mensagem semantica"
# Output: git commit -m "feat(auth): add JWT refresh token rotation"

Em 1 frase: Claude analisa o diff e gera commits semanticos (feat/fix/refactor) automaticamente.

2

Staging inteligente

O Claude prefere adicionar arquivos especificos ao staging (git add arquivo.ts) em vez de git add -A. Ele agrupa arquivos logicamente relacionados num mesmo commit, criando commits atomicos onde cada um representa uma unica mudanca coerente. Isso e critico: se precisar reverter, voce reverte uma unidade logica, nao um blob de mudancas misturadas.

Em 1 frase: Claude agrupa arquivos relacionados em commits atomicos, nunca usa git add -A cegamente.

3

Diff review automatizado

Antes de commitar, peca ao Claude: "revisa o diff atual". Ele roda git diff --staged, analisa as mudancas e reporta: bugs potenciais, inconsistencias de estilo, logica quebrada, imports faltando. E como ter um code reviewer instantaneo antes de cada commit.

diff-review.sh
# Pedir review do diff antes de commitar
git diff --staged | head -200
# Prompt: "analisa esse diff e me diz se tem algum problema"

Em 1 frase: Claude revisa o diff pre-commit e pega bugs antes que entrem no historico.

4

Merge conflicts

Merge conflicts sao o pesadelo do versionamento. Com Claude, voce pode pedir: "resolve os merge conflicts do arquivo X". Ele entende o contexto semantico de ambos os lados, nao faz uma resolucao mecanica (aceitar um ou outro), mas integra as intencoes de ambas as branches. Em conflitos complexos, ele pode perguntar qual comportamento voce prefere.

Em 1 frase: Claude resolve merge conflicts semanticamente, integrando a intencao de ambas as branches.

5

Branching strategy

Com IA no workflow, branches curtas funcionam melhor. O padrao: crie uma branch por feature/fix, deixe o Claude trabalhar nela, faca merge rapido. Branches de vida longa acumulam divergencia e dificultam o trabalho do agente. Prefira feat/add-auth que vive 1-2 dias a develop que acumula semanas.

Em 1 frase: branches curtas e focadas sao o melhor amigo do agente de IA.

6

.gitignore e seguranca

O Claude Code tem regras de seguranca embutidas: ele nunca comita arquivos .env, credentials ou chaves sem aviso explicito. Mas voce tambem deve manter seu .gitignore atualizado. Peca ao Claude para auditar: "verifica se tem algum segredo que nao esta no .gitignore". Prevencao e melhor que remediar um leak de credencial.

Nunca commitar

.env, *.pem, credentials.json, *.key, node_modules/, .claude/ (que contem worktrees temporarios). Se vazou, rotacione as chaves imediatamente.

Em 1 frase: .gitignore atualizado + Claude atento = zero vazamento de segredos.
